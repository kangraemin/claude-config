# HN Comment Generator

사용자가 HN 포스트/스레드를 붙여넣으면, 로컬 프로젝트를 스캔해서 실제 경험 기반의 댓글을 생성한다.

## 워크플로우

### 1. 원문 분석

사용자가 붙여넣은 HN 포스트에서 추출:
- 핵심 기술/주제 (언어, 프레임워크, 아키텍처, 개념)
- 글쓴이의 주장이나 설계 결정
- 스레드에서 아직 다뤄지지 않은 빈틈

답글 요청이면 특정 댓글의 맥락도 파악.

### 2. 프로젝트 스캔

`~/programming/` 하위 프로젝트들을 탐색해서 원문과 관련된 기술 경험을 찾는다.

스캔 전략:
1. 원문의 핵심 키워드로 프로젝트 매칭 (package.json, requirements.txt, build.gradle 등)
2. 매칭된 프로젝트의 코드를 읽어서 구체적인 경험 포인트 추출
3. 단순 "이 기술 써봤다"가 아니라, 실제로 부딪힌 문제나 설계 결정을 찾는다

주요 프로젝트 맵 (빠른 참조):
- **claudeInspector**: Electron 앱, Node.js, Claude API, IPC, 프록시 서버
- **coinbot**: Python, ccxt, 암호화폐 트레이딩 봇, 기술적 분석
- **stock-bot**: Python, yfinance, 주식 분석
- **slack-claude-bot**: Python, FastAPI, Slack Bot, Claude API 연동
- **ai-paper**: Node.js, 논문/아티클 수집기
- **gstack**: Claude Code 스킬 + headless browser 도구
- **RunnersHi**: Android (Kotlin/Gradle), 러닝 앱
- **stash-android**: Android 앱
- **game**: Electron, idle tycoon 게임
- **mcp**: Python, MCP 서버
- **seochogu**: Python, 이미지 처리
- **kangraemin.github.io**: 기술 블로그 (Jekyll)
- **area-insight**: 부동산/지역 분석
- **chart**: 차트 관련
- **fridge-app/server**: 풀스택 앱

관련 프로젝트가 없으면 억지로 끼워넣지 않는다. 그냥 기술적 관찰이나 질문으로 댓글을 쓴다.

### 3. 댓글 작성

#### 톤과 스타일

HN 댓글은 영어로 쓴다. `~/.claude/rules/writing-style.md`의 영어 문체를 따른다:

- 짧은 문장. 한 문장에 한 생각.
- `wrote`, `tried`, `broke`, `shipped` — `implemented`, `leveraged`, `utilized` 금지
- "amazing", "great work", "love this" 같은 빈 칭찬 금지
- 과장 금지. 담담하게.
- "이상했다 → 알고 보니" 구조가 자연스러움

#### 댓글 구조

좋은 HN 댓글의 패턴:

**경험 공유형** (프로젝트 매칭 있을 때):
```
[짧은 맥락 — 내가 뭘 만들면서 이걸 겪었는지]
[구체적 경험 — 어떤 문제/결정/발견]
[질문 또는 관찰 — 원글과 연결]
```

**기술적 질문형** (프로젝트 매칭 없을 때):
```
[원글의 특정 포인트 짚기]
[왜 궁금한지 한 줄]
[구체적 질문]
```

**답글형** (특정 댓글에 답할 때):
```
[상대 의견에 대한 직접 반응]
[내 경험이나 다른 관점]
[후속 질문 (선택)]
```

#### 금지사항

- AI가 쓴 티 나는 표현: "That's a great point", "This is interesting because", "I'd love to hear more about"
- 빈 칭찬으로 시작하기: "Great project!", "Nice work!"
- 너무 긴 댓글. 3-5문장이 적당.
- 관련 없는 프로젝트 억지로 언급하기
- 리스트/불릿 포인트 사용 (HN 댓글은 산문체)

#### 자기검열

댓글 생성 후 확인:
- [ ] 실제 프로젝트 경험이 구체적인가? (프로젝트명은 안 써도 되지만, 경험은 구체적이어야)
- [ ] HN에서 "AI slop"으로 flag 당할 표현이 없는가?
- [ ] 3-5문장 이내인가?
- [ ] 질문으로 끝나는가? (가능하면)
- [ ] 읽었을 때 진짜 개발자가 쓴 것 같은가?

### 4. 출력

반드시 아래 순서로 출력한다:

#### 4-1. 원문 브리핑 (한국어)

사용자가 원문을 이해할 수 있도록 한국어로 간단히 설명한다:

```
**글 요약**: [이 글이 뭔지 2-3줄 설명. 기술적 배경 포함]
**핵심 포인트**: [글쓴이의 주요 주장/설계 결정 1-2개]
**댓글 각도**: [왜 이 각도로 댓글을 썼는지, 어떤 프로젝트 경험과 연결되는지]
```

기술 용어는 괄호로 짧게 부연. 사용자가 글을 안 읽어도 맥락을 잡을 수 있어야 한다.

#### 4-2. 댓글 (영어)

인용 블록으로 보여준다. 사용자가 바로 복붙할 수 있게.

#### 4-3. 댓글 번역 (한국어)

댓글 영어 원문 바로 아래에 한국어 번역을 보여준다. 사용자가 내가 뭘 쓰는지 바로 파악할 수 있게.
자연스러운 한국어로. 직역 금지.

여러 각도가 가능하면 1개만 추천. 사용자가 다른 각도를 원하면 그때 추가로.
