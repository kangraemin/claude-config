# /qa-check

프로젝트 전체를 대상으로 잠재적 버그, 보안 취약점, 설정 문제, 의존성 이슈를 종합 점검한다.
`/review`가 변경된 코드만 보는 것과 달리, 이 스킬은 프로젝트 전체를 스캔한다.

---

## 실행 흐름

### Step 1: 프로젝트 파악

1. 프로젝트 루트에서 언어/프레임워크 감지:
   - `package.json`, `tsconfig.json` → Node.js/TypeScript
   - `build.gradle`, `pom.xml` → Java/Kotlin
   - `Package.swift`, `*.xcodeproj` → Swift/iOS
   - `Cargo.toml` → Rust
   - `go.mod` → Go
   - `setup.py`, `pyproject.toml`, `requirements.txt` → Python
2. `DEVELOPMENT_GUIDE.md`, `CLAUDE.md`, `.claude/rules/` 등 프로젝트 컨벤션 파일이 있으면 읽는다.
3. 프로젝트 구조를 빠르게 파악한다 (주요 디렉토리, 진입점, 설정 파일).

### Step 2: 점검 수행

아래 5개 카테고리를 순서대로 점검한다. 각 카테고리에서 주요 파일을 실제로 읽고 분석한다.

#### 2-1. 코드 로직 버그

핵심 비즈니스 로직 파일을 중심으로 탐색:
- null/undefined 처리 누락
- off-by-one 에러
- 경쟁 조건 (race condition)
- 잘못된 조건문/분기
- 에러 핸들링 누락 (빈 catch, 예외 무시)
- 무한 루프 가능성
- 타입 불일치

#### 2-2. 보안 취약점

- 하드코딩된 비밀키/토큰/비밀번호
- SQL/NoSQL 인젝션 가능성
- XSS, CSRF 위험
- 인증/인가 검증 누락
- 민감정보 로깅
- 안전하지 않은 역직렬화
- `.env`, 인증서 파일이 git 추적에 포함되었는지

#### 2-3. 설정/환경 문제

- 필수 환경변수가 문서화되지 않았거나 기본값 없이 사용
- 설정 파일 간 불일치 (dev vs prod)
- 잘못된 파일 경로/URL 참조
- 포트 충돌 가능성
- `.gitignore`에 빠진 항목

#### 2-4. 의존성 문제

- 패키지 매니저 lock 파일 존재 여부
- deprecated 패키지 사용 (알려진 것 위주)
- 버전 범위가 너무 넓은 의존성
- 사용하지 않는 의존성 (import 없이 선언만)
- 보안 취약점이 알려진 패키지 버전

#### 2-5. 코드 품질

- 데드 코드 (사용되지 않는 함수/변수/파일)
- 복붙 코드 (중복 로직)
- TODO/FIXME/HACK 주석 목록
- 과도하게 복잡한 함수 (깊은 중첩, 긴 함수)

### Step 3: 리포트 생성

점검 결과를 터미널에 출력하고, 프로젝트 루트에 `QA-REPORT.md`로 저장한다.

**리포트 포맷:**

```markdown
# QA Check Report

> 점검일: YYYY-MM-DD HH:MM
> 프로젝트: <프로젝트 이름>
> 스택: <감지된 언어/프레임워크>

## 요약

| 카테고리 | Critical | Important | Minor |
|----------|----------|-----------|-------|
| 코드 로직 | N | N | N |
| 보안 | N | N | N |
| 설정/환경 | N | N | N |
| 의존성 | N | N | N |
| 코드 품질 | N | N | N |
| **합계** | **N** | **N** | **N** |

## Critical 이슈

### [C-1] 이슈 제목
- **파일**: `경로/파일명:라인`
- **문제**: 무엇이 잘못되었는지
- **영향**: 어떤 문제가 발생할 수 있는지
- **수정 제안**: 구체적인 해결 방법

## Important 이슈

### [I-1] 이슈 제목
- **파일**: `경로/파일명:라인`
- **문제**: ...
- **수정 제안**: ...

## Minor 이슈

### [M-1] 이슈 제목
- **파일**: `경로/파일명:라인`
- **내용**: ...

## TODO/FIXME 목록

- `파일:라인` — 내용

## 점검하지 않은 영역

- 점검 범위 밖이었던 부분 (예: 외부 서비스 연동, 인프라 설정)
```

### Step 4: 안내

```
📋 QA 점검 완료. `QA-REPORT.md`에 저장했습니다.
Critical: N건 / Important: N건 / Minor: N건
```

Critical 이슈가 있으면 즉시 수정할지 물어본다.

---

## 심각도 기준

- **Critical**: 런타임 크래시, 데이터 손실, 보안 취약점 → 반드시 수정
- **Important**: 잠재적 버그, 에러 핸들링 누락, 성능 문제 → 수정 권장
- **Minor**: 코드 품질, 네이밍, 데드 코드 → 선택

## 점검 원칙

- 확실한 것만 보고한다. 추측성 이슈는 "가능성"을 명시한다.
- 파일을 실제로 읽고 분석한다. 파일명만 보고 판단하지 않는다.
- 프로젝트 규모가 크면 핵심 파일(진입점, 라우터, 모델, 설정)을 우선 점검한다.
- 이미 테스트로 커버된 부분은 낮은 우선순위로.
- 프로젝트 컨벤션을 존중한다. 컨벤션에 맞는 코드를 이슈로 잡지 않는다.
