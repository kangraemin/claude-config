# Review Rules

코드 리뷰 시 따르는 규칙.

---

## 1. 리뷰 관점 (우선순위 순)

1. **버그/로직 오류**: 잘못된 조건, off-by-one, null 처리 누락, 경쟁 조건
2. **보안 취약점**: 인젝션, 인증 우회, 민감정보 노출, 권한 검증 누락
3. **에러 핸들링 누락**: 예외 무시, 빈 catch, 실패 시 상태 불일치
4. **성능 문제**: N+1 쿼리, 불필요한 반복, 메모리 누수, 무한 루프 가능성
5. **설계/구조**: 단일 책임 위반, 의존성 방향, 추상화 수준, 중복 코드
6. **테스트 커버리지**: 핵심 로직 테스트 누락, 엣지 케이스 미검증
7. **네이밍/가독성**: 모호한 이름, 과도한 중첩, 매직 넘버

## 2. 심각도 분류

- **Critical**: 버그, 보안, 데이터 손실 → 반드시 수정
- **Important**: 설계, 에러 핸들링, 성능 → 수정 권장
- **Minor**: 네이밍, 스타일, 개선 제안 → 선택

## 3. 리뷰 원칙

- **변경된 부분만 리뷰한다.** 변경되지 않은 코드에 대한 지적 금지.
- **대안을 제시한다.** "이렇게 하면 안 된다"만 말하지 말고 구체적 대안 코드를 보여준다.
- **잘된 부분도 언급한다.** 좋은 설계, 깔끔한 처리는 인정한다.
- **확실한 것만 지적한다.** 추측으로 문제를 제기하지 않는다.
- **프로젝트 컨벤션 기준으로 판단한다.** 개인 스타일 강요 금지.
